# Team - 독립 에이전트 팀 병렬 실행

사용자 요청을 분석하고, **공식 Agent Teams**(독립 Claude Code 세션)을 자동 구성하여 병렬 실행한다.
어떤 프로젝트, 어떤 기술 스택에서든 동작한다. 모든 커뮤니케이션은 한국어.

> **중요**: 이 스킬은 서브에이전트(`Agent` 도구)가 아닌, 공식 Agent Teams 기능을 사용한다.
> 각 팀원은 독립 Claude Code 세션으로 실행되며, 서로 직접 메시지를 주고받을 수 있다.
> 토큰 사용량은 팀원 수에 비례하여 증가한다. 3-5명이 적정 규모이며, 팀원당 5-6개 태스크가 권장된다.

사용자 요청: $ARGUMENTS

- `$ARGUMENTS` 비어있으면 → AskUserQuestion으로 질문
- `--auto` 포함 시 → Phase 1 사용자 확인 생략, Phase 4 리드 통합 `/simplify`도 확인 없이 자동 실행 (팀원 개별 `/simplify`는 항상 자동)
- `--plan <path>` 포함 시 → Phase 1-1(계획서 생성) 스킵, 해당 경로의 계획서를 로드하여 Phase 1-2(작업 분해)부터 진행. Phase 0(분석)은 유지.
- `--plan`과 `--auto`는 조합 가능: 계획서 로드 + 사용자 확인 생략.

## Phase 0: 분석

**목표**: 환경 확인 + 프로젝트 파악 + 작업 유형/규모 판단

### 0-0. Agent Teams 활성화 확인

> **공식 문서**: https://code.claude.com/docs/en/agent-teams

**1단계: 버전 확인**
Claude Code v2.1.32 이상 필요. `claude --version`으로 확인.

**2단계: 환경변수 확인**
`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` 환경변수가 `1`로 설정되어 있는지 확인.

```bash
echo $CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS
cat ~/.claude/settings.json | grep AGENT_TEAMS
```

미설정 시 → 사용자에게 활성화 여부와 설정 범위를 질문:
```
Agent Teams 기능이 비활성화 상태입니다. 활성화할까요?

1. 활성화 (권장)
2. 작업 취소
```

사용자가 1번(활성화)을 선택하면 → 설정 범위를 질문:
```
어디에 설정할까요?
a. 글로벌 설정 (~/.claude/settings.json) — 모든 프로젝트에 적용
b. 프로젝트 설정 (.claude/settings.json) — 현재 프로젝트에만 적용
```

사용자 선택에 따라 해당 settings.json 파일을 직접 수정한다:
- **a (글로벌)**: 대상 파일 → `~/.claude/settings.json`
- **b (프로젝트)**: 대상 파일 → 프로젝트 루트의 `.claude/settings.json`

**실행 절차** (반드시 아래 순서대로 수행):
1. `Read` 도구로 대상 settings.json 파일을 읽는다 (파일이 없으면 5번으로)
2. 파일 내용에서 `"env"` 객체를 찾는다
3. `"env"` 객체가 있으면 → `Edit` 도구로 `"env": {` 뒤에 `"CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"` 를 추가
4. `"env"` 객체가 없으면 → `Edit` 도구로 최상위 `{` 뒤에 `"env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" },` 를 추가
5. 파일 자체가 없으면 → `Write` 도구로 새로 생성:
   ```json
   {
     "env": {
       "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
     }
   }
   ```
6. 수정 후 `Read` 도구로 파일을 다시 읽어 정상 반영되었는지 확인
7. 사용자에게 "설정 완료" 메시지 출력 후 바로 3단계로 진행

> 공식 문서: https://code.claude.com/docs/en/agent-teams

**3단계: 디스플레이 모드 확인 (선택)**
- `which tmux`로 설치 확인 (split-pane 모드용)
- 기본은 `"auto"` (tmux 세션 안이면 split, 아니면 in-process)
- `teammateMode` 설정: `"in-process"` (단일 터미널) / `"tmux"` (분할 패널)

**4단계: 환경변수 외 문제가 있을 때 사용자 확인**

> **절대 서브에이전트(`Agent` 도구)로 자동 대체하지 않는다.**

환경변수는 2단계에서 자동 설정된다. 그 외 문제(버전 미달 등)가 있을 때만 아래를 표시:

```
Agent Teams 환경이 준비되지 않았습니다.

현재 상태:
- 환경변수: {설정됨}
- 버전: {현재 버전} (필요: v2.1.32+) ← 버전 미달 시 표시
- tmux: {설치됨/미설치} (split-pane 모드용, 선택)

선택지:
1. 문제 해결 후 다시 시도 (권장)
2. 서브에이전트(Agent 도구)로 대체 실행 — 팀원 간 소통 불가, 독립 세션 아님
3. 작업 취소
```

사용자가 명시적으로 "2번" 또는 "서브에이전트로"를 선택한 경우에만 대체한다.

### 0-1. 규모 판단

| 규모 | 팀원 수 | 탐색 | 계획서 |
|------|---------|------|--------|
| 소규모 | 2명 이하 | 루트 설정 1~2개 | 생략 |
| 중규모 | 2~3명 | 표준 탐색 | `/feature-planner`로 생성 |
| 대규모 | 4~5명 (최대 5명) | 전체 탐색 | `/feature-planner`로 생성 (필수) |

### 0-2. 프로젝트 탐색

존재하는 파일만 탐색:
```
CLAUDE.md, package.json, tsconfig.json, pyproject.toml, setup.py,
requirements.txt, Cargo.toml, go.mod, build.gradle, pom.xml, Gemfile,
pubspec.yaml, Podfile, docker-compose.yml, .eslintrc*, .prettierrc*
```

- **모노레포 감지**: `workspaces`, `packages/`, `apps/` 존재 시 서브 디렉토리별 파악
- **프로젝트 파일 없음**: 사용자에게 질문

탐색 결과 정리:
```
프로젝트: {이름} | 루트: {경로} | 구조: {단일/모노레포}
스택: {언어, 프레임워크}
테스트: {커맨드 또는 "미설정"} | 린트: {커맨드 또는 "미설정"} | 빌드: {커맨드 또는 "미설정"}
컨벤션: {CLAUDE.md 핵심 3줄 요약 또는 "없음"}
```

### 0-3. 인프라 확인 (중/대규모)

- **테스트 인프라**: jest.config, pytest.ini, test/ 등 → 없으면 TDD 미적용 플래그
- **의존성 설치**: node_modules/, .venv/ 등 → 없으면 설치 필요 플래그
- **Git 상태**: `git status --short` → uncommitted 있으면 경고 + stash 권장

### 0-4. 작업 유형 판별

| 유형 | 전략 |
|------|------|
| **리서치** | 읽기 중심, TDD/검증 없음 |
| **구현** | 테스트 인프라 있으면 TDD |
| **버그 수정** | 조사 → 수정 순차 |
| **리팩토링** | 분석 → 구현 순차 |
| **문서화** | 병렬 작성, TDD/검증 없음 |
| **인프라** | 설정 전문 |
| **복합** | 유형별 혼합 |

**TDD 적용 조건**: 구현/리팩토링 + 테스트 인프라 존재 + 중규모 이상.

## Phase 1: 계획 + 분해 + 확인

**목표**: `/feature-planner`로 전체 계획 수립 → 팀원 단위로 분해 → 사용자 확인

### 1-1. 계획서 생성 (중/대규모 구현)

**`--plan <path>`가 전달된 경우**:
- 해당 경로의 파일 존재를 확인한다. 파일을 읽을 수 없으면 사용자에게 경로 확인을 요청하고 대기한다.
- `/feature-planner` 호출을 스킵한다.
- 계획서를 읽어 Phase/태스크 구조를 파악한다.
- Phase 1-2(작업 분해)로 바로 진행한다.

**그 외**:
리드가 `/feature-planner`를 사용하여 전체 구현 계획서를 생성한다.
- `docs/plans/PLAN_team-{키워드}.md`에 저장
- Phase별 태스크, Quality Gate, 성공 기준 포함
- 계획서의 각 Phase/태스크를 팀원에게 할당할 단위로 활용

소규모 또는 리서치/문서화 유형은 계획서 생략. 단, 계획서가 없더라도 리드가 각 팀원의 spawn 프롬프트에 **간이 작업 명세**(목표, 구체적 태스크 목록, 완료 기준)를 인라인으로 작성하여 전달한다.

### 1-2. 작업 분해

각 작업에 대해 결정:
- **작업명** (한국어) / **팀원명** (kebab-case)
- **역할**: 구현 / 리서치 / 리뷰 등
- **실행**: 병렬 또는 순차 (의존성 기반)
- **담당 파일 범위** (필수)
- **계획서 섹션**: 해당 팀원이 참고할 계획서의 Phase/태스크 번호

**파일 충돌 방지**:
- 팀원마다 담당 파일/디렉토리를 명시적으로 분리
- 공유 파일(`__init__.py`, `package.json` 등)은 한 팀원에게만 할당
- 겹치면 순차로 전환하거나 한 팀원이 전담

### 1-3. 사용자 확인

분해 결과를 표로 제시:

```
| # | 작업명 | 팀원명 | 역할 | 담당 범위 | 계획서 Phase | 의존성 |
|---|--------|--------|------|-----------|-------------|--------|
| 1 | ...    | ...    | 구현 | src/xxx/  | Phase 1     | 없음   |
| 2 | ...    | ...    | 구현 | src/yyy/  | Phase 2     | #1     |

이대로 진행할까요?
```

`--auto` 모드: 표는 보여주되 확인 없이 바로 실행.

## Phase 2: 팀 생성 + 실행

**목표**: Agent Teams로 독립 팀원 생성 + 태스크 할당 + 실행

> **핵심**: `Agent` 도구(서브에이전트)를 사용하지 않는다.
> `TeamCreate` → `TaskCreate` → 팀원 spawn 순서로 진행한다.
> 각 팀원은 **독립 Claude Code 세션**이다.

### 2-1. 팀 생성

`TeamCreate` 호출 — 이름: `{키워드}-{MMDD-HHmm}` (충돌 방지)

### 2-2. 태스크 등록

`TaskCreate`로 각 작업 등록:
- 명확한 제목과 설명
- `addBlockedBy`로 의존 관계 설정
- 태스크 설명에 담당 파일 범위 명시

### 2-3. 팀원 spawn

각 팀원을 독립 세션으로 생성한다. spawn 프롬프트에 아래 내용을 포함한다.
팀원은 독립 세션이므로 리드의 대화 히스토리를 상속하지 않는다 — 필요한 모든 컨텍스트를 프롬프트에 담아야 한다.

#### 공통 (모든 팀원)

> 계획서가 있는 경우 `## 리드 계획서` 섹션을 포함한다.
> 계획서가 없는 경우 (소규모, 리서치/문서화) 해당 섹션을 생략하고 `## 작업`에 충분한 컨텍스트를 담는다.

```
당신은 '{팀원명}' 팀원. {팀이름} 팀 소속.

## 프로젝트
{Phase 0에서 정리한 프로젝트 컨텍스트}

## 작업
Task #{번호}: {작업명}
{구체적인 내용 — 무엇을 구현/조사/작성해야 하는지 상세히}

## 리드 계획서 (계획서가 있을 때만 포함)
아래는 리드가 수립한 전체 계획에서 이 작업에 해당하는 섹션이다. 참고하되, 아래 "구현 계획 수립" 단계에서 자신만의 구체적 계획을 세워라.

{계획서에서 해당 팀원 담당 Phase/태스크 전체 내용 복사}

## 구현 계획 수립 (구현 작업 필수)
코딩을 시작하기 전에 반드시 아래 형식으로 구현 계획을 먼저 세워라.
계획을 세운 후 team-lead에게 SendMessage로 공유하고, 이상 없으면 바로 진행한다.
(리드의 승인을 기다릴 필요 없음 — 공유 목적은 방향 확인이며, 문제가 있으면 리드가 연락한다)

구현 계획 형식:
1. **구현 순서**: 어떤 파일/모듈을 어떤 순서로 작업할지
2. **핵심 설계 결정**: 자료구조, 인터페이스, 패턴 선택
3. **완료 기준**: 무엇이 되면 "끝"인지 (테스트 통과, UI 동작 등)
4. **리스크**: 어려울 수 있는 부분, 다른 팀원 작업과 겹칠 수 있는 부분

계획은 간결하게 — 10줄 이내로 충분하다. 과도한 플래닝보다 빠른 착수가 중요하다.

## 파일 범위
수정 가능: {목록}
그 외 파일 수정 금지.

## 코드 리뷰 (구현 작업만)
모든 구현 작업이 끝나면 반드시 `/simplify`를 실행하여 코드 리뷰를 수행하라.
/simplify는 코드 재사용, 품질, 효율성을 검토하고 문제를 직접 수정한다.
완료 기준 충족 + /simplify 완료 후에만 태스크를 completed로 변경하라.

## 완료
TaskUpdate로 태스크를 completed로 변경.
메시지에 포함: 구현 계획 대비 결과, 수정 파일 목록, /simplify 결과, 이슈.

## 소통
- 다른 팀원에게 직접 SendMessage 가능
- 문제 발생 시 team-lead에게 SendMessage로 보고
- 에러 무시 금지
```

#### 구현 팀원 추가 (TDD 적용 시만)

```
## TDD
RED → GREEN → REFACTOR. 테스트 커맨드: {커맨드}
구현 계획의 "구현 순서"에 테스트 작성 단계를 포함하라.

## 완료 전 체크
빌드 에러 없음 / 테스트 통과 / 기존 기능 영향 없음 / /simplify 완료
```

#### 리서치 팀원 추가

```
## 출력
{파일 경로}에 마크다운 작성. 구조: 요약 → 상세 → 추천 → 참고.
(리서치 작업은 구현 계획 수립과 /simplify 불필요 — 바로 조사에 착수하라)
```

### 2-4. 실행 방식

- **병렬 태스크**: 의존성 없는 태스크는 팀원이 동시에 자율 claim
- **순차 태스크**: `addBlockedBy`로 선행 완료 후 자동 unblock → 팀원 claim
- 팀원은 태스크 완료 후 다음 미할당 태스크를 자율적으로 claim

---

## Phase 3: 모니터링

**목표**: 완료 추적 + 순차 실행 관리 + 에러 대응

### 3-1. 진행 추적

- `TaskList`로 전체 태스크 상태 확인
- 팀원 메시지 수신 시 사용자에게 한 줄 요약 전달
- `Shift+Down`으로 팀원 간 전환하여 진행 상태 확인
- 팀원이 구현 계획을 공유하면, 전체 계획서와의 정합성을 빠르게 확인
  → 문제 있으면 즉시 SendMessage로 방향 수정
  → 없으면 무응답 (팀원은 바로 진행)

### 3-2. 순차 실행 관리

선행 태스크 완료 시:
- `addBlockedBy` 의존성이 자동 해소됨
- 대기 중인 팀원이 자율적으로 unblock된 태스크를 claim
- 필요 시 SendMessage로 팀원에게 선행 결과 요약 전달

### 3-3. 에러 처리

- **경미** (일부 미완성): 사용자 알림 + 미완성 부분 새 태스크 생성
- **심각** (진행 불가): 사용자 알림 + shutdown_request + 선택지 제시 (재시도/수동/스킵)
- **의존 실패**: 후속 태스크 중단 + 영향 범위 보고

### 3-4. 무응답 대응

팀원이 idle인데 태스크 미완료 시:
1. SendMessage로 상태 확인
2. 무응답 → 사용자 보고 + shutdown_request + 대체 팀원 spawn 여부 확인

## Phase 4: 검증 + 통합 리뷰 + 리포트

### 4-1. 검증 (구현/리팩토링/버그수정만. 리서치/문서화는 4-3으로 건너뜀)

**파일 범위 검사** (git 저장소인 경우만):
`git diff --name-only`로 변경 파일 확인 → 담당 범위 밖 수정 있으면 사용자에게 보고 + revert 제안

**의존성**: 미설치 플래그 있으면 사용자 확인 후 설치 실행

**빌드 → 테스트 → 린트** 순서로 실행 (각각 설정이 있을 때만):
- 실패 시 에러 분석 → 직접 수정 또는 수정 팀원 배치

### 4-2. 리드 통합 코드 리뷰

모든 팀원 작업이 완료되면, **리드가 `/simplify`를 실행**하여 전체 변경사항에 대한 통합 리뷰를 수행한다.

각 팀원이 이미 개별적으로 `/simplify`를 실행했지만, 리드의 통합 리뷰는 다른 관점을 갖는다:
- **팀원 간 중복**: 서로 다른 팀원이 유사한 유틸리티/패턴을 독립적으로 만들었는지
- **통합 일관성**: 팀원들의 코드가 합쳐졌을 때 네이밍/스타일/추상화 수준이 일관적인지
- **인터페이스 정합성**: 팀원 A가 만든 함수를 팀원 B가 사용하는 부분의 계약이 맞는지

`/simplify` 결과를 리포트에 포함한다.

### 4-3. 팀원 종료

아직 실행 중인 팀원에게 shutdown_request 전송.
팀원이 현재 작업을 마무리한 후 종료됨.

### 4-4. 종합 리포트

```
## 팀 작업 완료 리포트

### 프로젝트: {이름} ({스택}) | 규모: {소/중/대}
### 팀: {팀이름} | 팀원: {N}명

### 결과
- 완료: N개 / 실패: N개 / 스킵: N개

### 검증 (해당 시)
- 빌드: 성공/실패
- 테스트: N통과/N실패 (또는 "미설정")
- 린트: 통과/경고N (또는 "미설정")

### 코드 리뷰 (/simplify)
- 팀원 개별 리뷰: {각 팀원 /simplify 결과 요약}
- 리드 통합 리뷰: {통합 /simplify 결과 요약}

### 작업별
#### 1. [작업명] (팀원: xxx)
- 결과: ...
- 파일: ...
- /simplify: {수정 사항 또는 "코드 깨끗함"}

### 이슈 (있으면)
- ...

### 다음 단계
- ...
```

2. `TeamDelete`로 팀 리소스 정리
